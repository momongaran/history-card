1. 概要

v3.2完了後の設計検討（doc/notes_v3_3_design.html）で確定した以下の変更をデータ・アプリに反映する。

  1. momentum（Eの解放の方向）をframeworkViewに導入

  5. B体系は既存カテゴリで十分、追加不要

2. momentum フィールド

2.1 定義

  • Eの解放のエネルギーがどの方向に放出されているかを記述する
  • accumulationの対概念: accumulation=過去方向、momentum=未来方向
  • Eの「結果」や「帰結」ではなく、解放の瞬間に内在する方向性
  • 意図的か構造的帰結かを問わない
  • 1イベントに1つのmomentum

2.2 データ形式

{
  "frameworkViewId": "FW_EV_p0593_01",
  "accumulation": { "relations": [...], "summary": "..." },
  "momentum": "王権を中心とした安定的統治が求められる方向へ"
}
  • frameworkViewの直下、accumulationと同レベルに配置
  • 型: string | null
  • null = 未生成（初回マイグレーション時）
  • 記述形式: 「〜の方向へ」で終わる15〜40文字程度の文

2.3 生成手段

scripts/generate_momentum.py により Claude API で全221件を生成。 scripts/merge_momentum.py で本体に統合。

2.4 逓増パターンとの関係

momentumの連鎖を時系列で追うことで、逓増（increasing returns）パターンが浮かび上がる。

類型特徴例
単線型方向が一貫、各イベントが同一方向のmomentumを持つ律令国家建設（593→701、約100年）
漸進型方向は一貫だが、当事者にその自覚がない中央統治→地方自律（939→1192、約250年）
収斂型複数の独立したmomentumが合流して1つの帰結に至る元禄文化（1590→1688、約100年）

3. F体系の拡張

3.1 Fの再定義

旧定義新定義
Fluctuation（揺らぎ）= 不確実性・危機F（起動因）= このイベントがこのタイミングで起きた理由
Fの頭文字の由来（Fluctuation）と実際のカテゴリ範囲にずれが生じるが、B/F/Eという記号体系として「第2層」の位置づけで機能させる。

3.2 F-PRGカテゴリの新設

カテゴリ性質例
F-UNC（既存）不確実性・危機型王位空位、災害発生、外圧急変
F-PRG（新規）上位プログラムの実行条件成立大化改新の税制が全国施行

F-PRGは「上位のイベントのEが下位のイベントのFとして作用する」構造を記述する。

3.3 v3.3での適用

sortKey 647（租庸調により農民負担が急増した）のF要素を F-UNC-00（未設定）から F-PRG-01 に変更。

taxonomy_v3_0_bfe.json にはF-PRGを今回追加しない（データ内のcategory文字列のみ変更）。taxonomyの更新は次回以降のF体系全体再設計時に行う。

4. 視点ルール

4.1 基本方針（案2）

層視点理由
B（背景）客観記述構造的背景は特定の誰かの視点に依存しない。Bのパターンマッチングを維持する
F（起動因）実行者視点「この人がこのタイミングで動いた理由」は実行者視点が自然
E（解放）実行者視点「この人が放出したエネルギーの方向」は実行者視点が自然

4.2 v3.3での適用

B不足6件の書き直しで案2を適用。該当イベントのF/Eラベルを実行者視点に更新。

### 1.7 背景深度と展開表示（Background Depth & Expansion）

#### 設計原則: 中間ノードは合成器である

中間ノードAは複数の構造的条件を「特定の形」に合成する。合成後の出力がターゲットCに到達するのであって、原料が直接Cに流れるわけではない。**したがってエッジの自動追加は行わない。**

例: N008（大和の任那出兵圧力）は5つの構造的条件（継体の脆弱性、豪族分立、半島情勢、任那権益縮小、磐井独自外交）を「出兵圧力」という特定の形に凝縮している。この凝縮された圧力がN009（磐井の乱）をtriggersするのであって、5つの条件が直接N009に作用しているわけではない。

#### 問題と解決

triggersエッジで単一入力しかないノードは、UI上で背景が薄く見える。しかしこれはエッジの問題ではなくUIの問題である。解決策はトリガーソースの背景ツリーを**展開表示**すること。

#### 検証スクリプト（読み取り専用）

`scripts/propagate_backgrounds.py` はエッジの追加・変更を一切行わない。背景が薄く見えるノードを検出し、トリガーソースの背景ツリーを最大3段の深さで表示する。

```bash
python3 scripts/propagate_backgrounds.py           # レポート表示
python3 scripts/propagate_backgrounds.py --json     # JSON出力（UI連携用）
```

v2データ: 10ノードが展開表示の候補。

#### 改訂経緯

1. 初版: amplifies経由伝播+depth=2でエッジを自動追加 → 208→381エッジ（過剰接続）
2. 改訂: triggers経由のみ+既存入力2本以下 → 208→238エッジ（双子状態を生成）
3. 最終: **エッジ追加自体を廃止**。中間ノードは合成器であり、原料を直接流すべきではない。UIの展開表示で解決する。

## 3. UI画面構成

### 3.1 因果フレーム図鑑（demo_catalog.html）

インスタンスビュー。EnergyAbsグラフの全ノードをクラスタ→深度順にブロック表示。

- **ブロック**: 枠色=fwType色、内部にref-chip（入力元）・ラベル・概要・抽象パターン・アクター・出力先
- **SVG線**: ref-chip⇔ソースブロック間の同一性リンク（点線）
- **コントロール行1**: roleフィルタ（amplifies/triggers/transforms/sustains）、凡例トグル
- **コントロール行2**: 意味領域フィルタ、fwTypeフィルタ、逆引き検索
- **凡例**: フローティング、デフォルト非表示

### 3.2 抽象パターン図鑑（demo_pattern_catalog.html）

パターンビュー。エッジをfwType遷移型でグルーピングし、抽象パターンチェーンとして表示。

- **セクションヘッダ**: fwType遷移型（例: ⚡→◆）、パターン数・件数
- **パターンカード**: fw-icon（26px角、色付き、ホバーでラベル）→ パターン文（{スロット}にslot-icon付き）
- **インスタンス折りたたみ**: 各パターンの具体的ノードペア一覧
- **フィルタ**: fwType別絞り込み

### 3.3 スロット分類一覧（demo_slot_taxonomy.html）

分類ビュー。128語のスロット語彙を7大分類・27小分類で一覧表示。

- **サマリテーブル**: 大分類×小分類の語数・使用回数
- **カテゴリセクション**: SVGアイコン付きヘッダ
- **用語カード**: クリックで使用箇所展開（該当スロットをハイライト）

## 5. 型レベル再集約の分析結果（v1時点の予備調査）

### 5.1 スロット型シグネチャによる集約

70個のユニークパターンをスロットの大分類型の並びだけで集約すると **36シグネチャ** になる。上位6シグネチャで全体の49%（34/70件）をカバー。

| シグネチャ | 件数 | 例 |
|---|---|---|
| ●主体→□構造 | 11 | {勝者}が{主導体制}を確立する |
| ●主体→●主体 | 10 | {新秩序派}が{旧秩序派}を武力排除する |
| ●主体→◇力学 | 4 | {指導者}の死が{後継争い}の空白を生む |
| ●主体（単独） | 3 | {地方}が叛乱を起こす |
| ●主体→≈変動 | 3 | {王権}が{対外関与}の伝統を持続する |
| ▬資源→▬資源 | 3 | {人口}が{制度の容量}を超過する |

### 5.2 作用型（動詞の方向性）による二次集約

スロット型シグネチャが同じでも動詞の方向性が異なるケースがある（例: 「確立する」vs「打倒する」）。動詞を作用型に正規化することで、実質 **20〜25個の集約パターン** に収まる見込み。

作用型の候補（6〜8種）:

| 作用型 | 含まれる動詞 | 例 |
|---|---|---|
| 獲得 | 確立/掌握/樹立/完成/独占/建設 | ●が□を獲得する |
| 破壊 | 打倒/解体/崩壊/排除/滅ぼす | ●が□を破壊する |
| 蓄積 | 高まる/激化/拡大/超過 | ◇が蓄積する |
| 維持 | 持続/継承/定着 | ●が≈を維持する |
| 衝撃 | 揺さぶる/襲う/仕掛ける | ●が□に衝撃を与える |
| 逆転 | 自立化/凌駕/迂回 | ●が●を逆転する |
| 動員 | 動員/利用/呼応 | ●が●を動員する |
| 変容 | 転換/侵食/形骸化/乖離 | ▬が▬を変容させる |

### 5.3 次バージョンの実装方針

1. 作用型を確定し `slot_taxonomy.json` に追加
2. 各パターンに `typeSignature`（スロット型並び）と `actionType`（作用型）を付与
3. パターン図鑑を集約パターンで再グルーピング
4. 世界史展開時に再分類を見直す前提で設計

### 6.7 議論メモ: 知識吸収の動的安定性問題 (2026-03-22)

> 課題D-6の背景議論。資本主義テキスト吸収の初回実証で浮上した根本的問題。

#### 問題の所在

知識吸収プロセス（doc/knowledge_absorption_process.md）の品質指標5種は全て**静的スナップショット**である。
マージ後の状態が「エントロピー適度に増加、構造歪みゼロ」であっても、**内部に蓄積された歪みは計測できない**。

具体例: 資本主義テキスト吸収で W037b(重商主義) を W029, W025b の2本で接続した。
この瞬間は安定に見えるが、次に「宗教改革と経済の関係」ソースが入った場合、
W032→W037b の接続が必要になり W037b 周辺のエッジ構造が窮屈になる可能性がある。
**現在の計測ではこの「将来の窮屈さ」が見えない。**

#### メタ的自己言及

この問題はフレームワーク自身の概念で記述できる:

- 現在のグラフ = **institution**（安定した構造）
- 内部歪み = **drift**（表面化していない構造的問題）
- 将来の吸収困難 = collapse/reform の予兆
- エントロピー最大に見えるが内部に歪みが蓄積 = **「見かけの安定期」の古典的パターン**

すなわち「エントロピーが最も上がるマージが最良」ではない。エントロピーは体温計であり、
**変化の仕方**に意味がある。高熱も低体温も問題であり、「平熱の範囲で適度に上がった」が健全。

#### 動的評価に必要なアプローチ（未実装）

| アプローチ | 概要 | 材料力学の対応 |
|---|---|---|
| **摂動応答テスト** | 仮想的にノード/エッジを1つ追加・削除した場合に、何本のエッジ再構成が要求されるか | 応力テスト |
| **将来吸収余力** | 各ノードの次数余裕、密度偏りの余白。「この領域にあと何ノード追加できるか」 | 弾性限界 |
| **因果的整合性チェック** | A→B amplifies, B→C amplifies なのに A→C が無い場合、それは意図的省略か暗黙の歪みか | 残留応力検出 |
| **多ソース交差検証** | 異なるソースが同じノードに矛盾する因果を主張した場合の検出・記録 | 疲労試験 |

#### 現実的な検証戦略

上記の定義は「何本まで許容か」「どこからが歪みか」の基準が必要だが、
これは**データが蓄積されて初めて校正可能**。

- 資本主義テキスト（第1例）→ 次のソース吸収時に「前回の吸収が堅牢だったか」を実証的に検証
- 複数ソース吸収後に、歪み蓄積の実例パターンを帰納的に抽出
- 基準値は演繹ではなく実例から校正する（vision_validation原則に準拠）

#### 結論

静的指標（エントロピー、構造歪み度等）は**必要条件のチェックには有用**だが、
**十分条件の保証にはならない**。動的安定性の評価は将来課題として残し、
吸収事例を蓄積する中で帰納的に基準を確立していく。

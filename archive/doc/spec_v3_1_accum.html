myHistory 仕様 v3.1（accumulation 追加）

実装確定版 — 2026-02-23

1. 概要

v3.0 の B/F/E 3レイヤー構造を維持したまま、B要素間の蓄積関係（accumulation）を各 frameworkView に追加する。 B要素は v3.0 では平坦なリストとして並んでいたが、v3.1 では「どの条件とどの条件が組み合わさってエネルギー蓄積になるか」を明示する。

同時に、B要素ラベルの質的改訂（行為記述 → 条件記述）とカテゴリ誤分類の修正も実施した。

2. カテゴリ体系（全分類コード）

taxonomy の正本は data/taxonomy_v3_0_bfe.json。v3.1 でカテゴリ体系自体の追加・削除はないが、51件のB要素で誤分類を修正した。

3. accumulation スキーマ

各 frameworkView に accumulation フィールドを追加。

{
  "accumulation": {
    "relations": [
      {
        "from": "EL_EV_p0527_01_1",
        "to": "EL_EV_p0527_01_2",
        "mechanism": "tension",
        "description": "資源基盤と社会構造が逆方向に作用し構造的緊張が蓄積された"
      }
    ],
    "summary": "権力配置と資源と社会構造の緊張が蓄積し事象に至った"
  }
}

3.1 フィールド定義

フィールド型説明
accumulationobject | nullB要素間の蓄積関係。B≤1 のFWでは null。
accumulation.relationsarrayB要素ペア間の関係。
relations[].fromstring起点B要素の elementId。当該FW内に存在すること。
relations[].tostring終点B要素の elementId。from と異なること。
relations[].mechanismstring蓄積メカニズム（5種のいずれか。2.2節参照）。
relations[].descriptionstring蓄積メカニズムの説明（日本語、20〜100文字目安）。
accumulation.summarystring全体の蓄積構造の要約（日本語、1文）。

3.2 mechanism 語彙（5種）

コード日本語定義
tension緊張二条件が逆方向に作用し構造的ストレスを生む
amplification増幅一条件が他を強化・激化させる
constraint制約一条件が選択肢を狭め危機に向かわせる
enabling促進一条件が他の作動前提を作る
convergence収斂独立した条件が同一の圧力点に集中する

4. B要素ラベル・カテゴリ改訂

4.1 ラベル改訂

行為記述（〜した、〜された）を条件記述（〜していた、〜であった、〜されていた）に修正。事象発生「前」の背景条件として読める文体に統一した。

修正前（例）修正後（例）
大伴金村が大連として政務を掌握した大伴金村が大連として政務を掌握していた
百済の聖明王が軍事同盟強化を図った百済の聖明王が軍事同盟強化を図っていた

既に条件記述（〜していた等）であったラベルは変更なし。

4.2 カテゴリ改訂

明らかな誤分類を修正。主な修正パターン:

  • 軍事関連ラベルが B-SOC に入っていたものを B-MIL に修正
  • 対外環境ラベルが B-NRM に入っていたものを B-EXT に修正
  • 権力掌握ラベルが B-INS に入っていたものを B-PWR に修正

6. 中間データ形式

生成スクリプトは中間ファイル accumulation_v3_1.json を出力し、マージスクリプトで本体に統合する。

{
  "version": "3.1",
  "generatedBy": "generate_accumulation_local.py",
  "model": "rule-based",
  "timestamp": "2026-02-23T12:34:00",
  "items": {
    "FW_EV_pXXXX_XX": {
      "accumulation": { "relations": [...], "summary": "..." },
      "revisions": [
        {
          "elementId": "EL_...",
          "labelRevised": "条件記述ラベル",
          "categoryRevised": "B-XXX",
          "subCategoryRevised": "B-XXX-NN"
        }
      ]
    }
  }
}

7. バリデーションルール

  • from / to が当該FWのB要素 elementId に存在すること
  • mechanism が5種の語彙に含まれること
  • self-reference がないこと（from != to）
  • B≥2 のFW全てに accumulation があること
  • B≤1 のFWでは accumulation: null
  • description 長: 20〜100文字（目安。多少の逸脱は警告扱い）
  • mechanism 分布が極端に偏っていないこと（単一種 > 60% なら要確認）

12. 検証方法

  • ブラウザで src/index.html を開き、既存表示が壊れていないこと。
  • 221イベントすべて表示されること。
  • 各行: B要素（複数）→ F要素（1つ）→ E要素（1つ）の順。
  • B要素ラベルが条件記述（〜していた等）になっていること。
  • コンソールエラーなし。
  • merge_accumulation.py のバリデーションがエラー0件で通ること。

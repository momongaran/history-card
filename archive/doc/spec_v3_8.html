myHistory 仕様 v3.8（統合版）

実装確定版 — 2026-02-24

1. 概要

v3.2完了後の設計検討（doc/notes_v3_3_design.html）で確定した以下の変更をデータ・アプリに反映する。

  1. momentum（Eの解放の方向）をframeworkViewに導入
  2. F体系の拡張: 「揺らぎ」→「起動因」に意味を拡張、F-PRGカテゴリ新設
  3. 視点ルール: B=客観記述、F/E=実行者視点（案2）
  4. B不足6件の補充: 既存カテゴリで対応
  5. B体系は既存カテゴリで十分、追加不要

2. momentum フィールド

2.1 定義

  • Eの解放のエネルギーがどの方向に放出されているかを記述する
  • accumulationの対概念: accumulation=過去方向、momentum=未来方向
  • Eの「結果」や「帰結」ではなく、解放の瞬間に内在する方向性
  • 意図的か構造的帰結かを問わない
  • 1イベントに1つのmomentum

2.2 データ形式

{
  "frameworkViewId": "FW_EV_p0593_01",
  "accumulation": { "relations": [...], "summary": "..." },
  "momentum": "王権を中心とした安定的統治が求められる方向へ"
}
  • frameworkViewの直下、accumulationと同レベルに配置
  • 型: string | null
  • null = 未生成（初回マイグレーション時）
  • 記述形式: 「〜の方向へ」で終わる15〜40文字程度の文

2.3 生成手段

scripts/generate_momentum.py により Claude API で全221件を生成。 scripts/merge_momentum.py で本体に統合。

2.4 逓増パターンとの関係

momentumの連鎖を時系列で追うことで、逓増（increasing returns）パターンが浮かび上がる。

類型特徴例
単線型方向が一貫、各イベントが同一方向のmomentumを持つ律令国家建設（593→701、約100年）
漸進型方向は一貫だが、当事者にその自覚がない中央統治→地方自律（939→1192、約250年）
収斂型複数の独立したmomentumが合流して1つの帰結に至る元禄文化（1590→1688、約100年）

3. F体系の拡張

3.1 Fの再定義

旧定義新定義
Fluctuation（揺らぎ）= 不確実性・危機F（起動因）= このイベントがこのタイミングで起きた理由
Fの頭文字の由来（Fluctuation）と実際のカテゴリ範囲にずれが生じるが、B/F/Eという記号体系として「第2層」の位置づけで機能させる。

3.2 F-PRGカテゴリの新設

カテゴリ性質例

F-PRGは「上位のイベントのEが下位のイベントのFとして作用する」構造を記述する。

13. v3.5 差分: F定義厳密化・parentFramework・origin・F-PRG廃止

実装確定版 — 2026-02-26

13.1 背景と理論的根拠

v3.4でF要素のカテゴリ体系は整備されたが、以下の問題が残っていた：

  1. F定義の曖昧さ: Fラベルの約54%がテンプレート文・B重複・E記述であり、「揺らぎ」の理論的定義が不明確だった。
  2. プログラム実行型イベントの扱い: 一部のイベント（~11件）はBに構造的エネルギーがなく、上位プログラムの実行段階に過ぎなかった。
  3. 帰属的進化の未表現: E（解放）が後続イベントのB（背景）になる再帰的構造を表現する手段がなかった。

13.2 F（Fluctuation / 揺らぎ）の厳密な定義

定義: Fとは、Bに内在する構造的勾配を顕在化させ、系を分岐状態へと押し込む具体的事象である。

3条件:

  1. 具体的事象である（抽象的傾向ではない）
  2. 分岐を引き起こす（Fなしには即座に同じ帰結に至らない）
  3. 単独では構造を説明できない（Bなしには事象は吸収される）

Fは原因ではない。結果でもない。背景でもない。Fは、分岐を現実化させる選択圧である。

相転移モデルとの対応

層物理的対応役割
B（Background）潜在的秩序勾配（過冷却状態）エネルギーを蓄積する構造的条件
F（Fluctuation）分岐刺激（振動・核）勾配を顕在化させる選択圧
E（Emancipation）勾配解放（相転移）構造変化の方向（momentum）

重要: Fは「臨界点に押し上げる」のではない。臨界性を作るのはBである。Fは既に存在していた勾配を選択状態にする刺激である。

Bのエネルギー要件

Bは単なる条件の列挙ではなく、構造的勾配（潜在エネルギー）を含むべきである。 エネルギー源の類型：

  • 構造的矛盾: 制度と実態の乖離、支配構造の不整合
  • 恐怖・危機感: 外圧への恐怖、植民地化の不安、体制崩壊の予感
  • 利害対立: 勢力間の緊張、資源配分の不均衡
  • 蓄積された圧力: 財政逼迫、社会不安の蓄積

逆算的記述（結果を知った上で後付けした圧力表現）は避ける。 当時の人間が実際に認識・体感していた構造的圧力を記述する。

13.3 parentFramework フィールド

221件中約11件は独立した相転移ではなく、上位フレームワークのEの展開過程（実行ユニット）である。 これらはBに固有の構造的エネルギーを持たず、上位FWからエネルギーを継承する。

データ構造

{
  "parentFramework": {
    "eventId": "EV_p0701_01",
    "relationship": "execution_unit"
  }
}

parentFrameworkはframeworkViewの直下に配置する。独立した構造型イベントでは省略（null）。

relationship の値

値意味
execution_unit上位FWのEの実行段階。Bのエネルギーは親から継承。

階層性

parentFrameworkは再帰的である。子が孫の親になれる。チェーンを辿ることで任意深度の階層が復元できる。

13.4 origin フィールド（帰属的進化）

B要素が先行イベントのEから再帰的に生成された場合、その出自を追跡する。 これは「帰属的進化」（E→Bの再帰的構造）の最小表現である。

データ構造

{
  "elementId": "EL_EV_p0647_01_1",
  "layer": "B",
  "category": "B-INS",
  "subCategory": "B-INS-02",
  "label": "大化改新により租庸調・班田収授が制度化され全国一律の徴税が始まっている",
  "origin": {
    "sourceEventId": "EV_p0645_01",
    "sourceLayer": "E"
  }
}

originはB要素のオプショナルフィールド。 存在しない場合は省略（従来互換）。全B要素に必須ではなく、E→B帰属が明確な場合にのみ記述する。

13.5 F-PRG カテゴリの廃止

v3.4で導入したF-PRG（上位プログラム）は廃止する。

理由

  1. 適用1件のみ（EV_p0647_01）
  2. parentFrameworkフィールドの導入により、「上位プログラムの実行」はデータ構造で表現可能になった
  3. F-PRGの1件はF-ACT-01（具体的行動: 全国施行の開始）に再分類可能
  4. F定義の純粋性が保たれる: Fは常に「選択圧」であり、プログラムへの従属物ではない

taxonomy変更

F-PRGカテゴリおよびF-PRG-01サブカテゴリをtaxonomyから削除する。

データ移行

EV_p0647_01のF要素: F-PRG/F-PRG-01 → F-ACT/F-ACT-01 に変更。

13.7 設計原則のまとめ

  • Bはエネルギーを持つ: 構造的勾配（矛盾・恐怖・対立・圧力）を含む。単なる条件の列挙ではない。
  • Fは選択圧: 原因でも結果でも背景でもない。分岐を現実化させる刺激。
  • 実行ユニットは親から継承: Bに固有のエネルギーがないイベントは parentFramework で親を参照。
  • 帰属的進化はoriginで追跡: E→Bの再帰的構造を最小限のフィールドで表現。
  • 逆算的記述の禁止: Bの圧力は当時の人間が認識していた構造的条件として記述する。

14. v3.6 差分: B/F定義拡張・文体基準

14.1 背景

v3.5のBのエネルギー要件（13.2）を以下のように拡張する。

  • エネルギー型（基本形）: 矛盾・恐怖・対立・圧力といった構造的勾配を持つ条件。
  • 土台型: それ自体はエネルギーを持たないが、エネルギー型Bが生じる前提となる構造的事実。origin フィールドを持つ場合にのみ使用する。

15.1 背景

15.2 F/E分離原則

原則: FはEの具体版であってはならない。Fが記述するのは「分岐を引き起こした刺激」であり、「分岐の結果」ではない。

禁止パターン

#パターン説明例（誤）

判定基準

テスト: 「Fを削除しても、Eが同じ内容を言っているか？」→ Yesなら重複。

正しいFは、Eとは独立した情報を持つ。Fがなければ「なぜこの構造変化が起きたのか」が分からなくなる。

16. 検討メモ: Bのエネルギー出自分類とEの粒度展開

未確定 — 2026-02-28 記録

以下は仕様化の要否を含め未確定の検討事項である。将来の設計判断のために記録する。

16.1 Bのエネルギー出自による分類（命令形・プログラム形）

現行のBは「構造的勾配がある」と規定するのみで、その勾配がどのような力学で実行を迫るかを区別していない。 少なくとも以下の3型が観察される：

型力学例
蓄積型（現行の典型）矛盾・圧力が臨界に達する荘園制の矛盾、外圧への危機感
命令型明示的ルール・権威・制度的義務・強制力が実行を迫る律令の施行、幕府の法度、暗黙のルールや文化的規範
プログラム型上位の計画・意思決定が下位の実行を駆動する大化改新→租庸調、明治政府→廃藩置県

命令型とプログラム型は重なる。プログラム型は「上位Eが命令型Bを生み出す」構造であり、 プログラム型は命令型の特殊ケースと見なせる可能性がある。

現行仕様との関係

  • プログラム型: parentFramework（execution_unit）で構造的に表現済み。
  • 命令型: 未表現。originフィールドの拡張（「このBの実行圧力はどこから来ているか」の記述）が候補。
  • 蓄積型: 現行のBエネルギー要件そのもの。originなし（自律的に蓄積された勾配）。

14.3の土台型B（エネルギーを持たない構造的前提）の議論とも関連する。 仕様化する場合は土台型との整合性を考慮する必要がある。

16.2 Eの粒度展開（展開形）

1つのEが粒度的に大きすぎる場合（例: 明治維新のE「近代国家体制への転換」は、 廃藩置県・地租改正・徴兵令・学制など複数のサブ構造変化から構成される）、 タスクの階層化のようにEをツリー状に展開する必要が生じうる。

現行仕様での対応状況

parentFrameworkのexecution_unitは、実質的にこの展開を表現している： 上位FWのEを複数の子FWに分割し、各子FWが独立したB/F/E構造を持つ。

明治維新 (1868)  ← 上位FW: E = 近代国家体制への転換
  ├→ 廃藩置県 (1871)    parentFramework → 1868
  ├→ 地租改正 (1873)    parentFramework → 1868
  ├→ 徴兵令 (1873)      parentFramework → 1868
  └→ 学制 (1872)        parentFramework → 1868

新しいrelationshipタイプ（例: decomposition）を追加する案もあるが、 execution_unitとの境界が曖昧になるリスクがある。 現時点では既存のparentFrameworkで対応し、データの充実（子FWの追加）で解決する方針を維持する。

未解決の論点

  • 兄弟FW間の順序制約（「廃藩置県の後に地租改正が来る必然性」）はparentFrameworkだけでは表現できない。momentumの連鎖で部分的にカバーされるが、明示的な順序関係の表現が将来必要になる可能性がある。
  • E内部をFWを増やさずにサブステップ分解する需要があるかは、現行205件の運用を通じて観察を続ける。

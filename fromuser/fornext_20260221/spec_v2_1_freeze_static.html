myHistory 仕様 v2.1 Freeze（静的UI / 揺らぎF必須1）

作り手に渡す確定版（Phase1）

1. このFreezeの狙い

  • 「確定した部分だけ」をFreezeし、作り手と表示側の共通認識を揃える。
  • 未確定（今回対象外）：グローバルレーン、6軸円環、動的演出（ホバー線/アニメ等）、対応リンクの描画ルール。

2. データモデル（FrameworkView）

2.1 1イベント = 1 FrameworkView

  • 各イベント eventId に対し、frameworkViews は必ず1つ存在する。

2.2 レイヤー（layer）

既存のレイヤー体系（C/P/R）を維持しつつ、揺らぎ専用レイヤー F を追加する。

layer意味（Freeze時の扱い）
CBackground（背景）。既存taxonomyのContext群（C-*）をそのまま継承する。
FFluctuation（揺らぎ）。必須1（Primary Fluctuation）。
RResult（結果）。既存taxonomyのResult群（R-*）をそのまま継承する。
PPath（経路）。データとして存在してよいが、Freeze静的UIの必須対象にはしない（表示してもしなくてもよい）。

2.3 必須制約（Freeze）

レイヤー必須数備考
C（背景）1以上既存資産を維持
F（揺らぎ）ちょうど1Primary Fluctuation。複数Fは将来拡張（未決定）
R（結果）1以上既存資産を維持

3. taxonomy（カテゴリ体系）

  • taxonomy_v2_1_freeze.json は、既存の C/P/R/L を切り落とさず維持した上で、F を追加する。
  • 揺らぎは旧 C-UNC から切り出し、F-UNC として移動する（サブカテゴリも移植）。

4. 理論注記：吸収能力 A(t)

吸収能力 A(t) は「背景（C）の状態変数」として仮定しうる。

ただし Phase1 Freeze では A(t) を独立要素としてデータ化・可視化しない（注意書きとして保持）。

将来、背景に変化を与えるフレームワーク（背景変化FW）として表現する可能性がある。

5. 対応リンク（correspondences）

  • リンクカテゴリ（L）は既存taxonomyを維持する。
  • Freeze静的UIでは「イベント↔要素を結ぶ線」の描画を行わない（表示しない）。
  • 理由：特に長期揺らぎ等は、直線的リンクだけで意味を確定させるのに無理が出やすいため。

6. UI仕様（静的Freeze）

  • 左：イベント行（年号/タイトル）
  • 右：当該イベントの要素バッジを横並びで表示
  • 表示順序：C → F → P（任意） → R
  • 動的演出：なし（ホバー線/アニメーションなし）
  • リンク線：描画しない

7. 配布物

  • framework_output_v2_1_freeze.json
